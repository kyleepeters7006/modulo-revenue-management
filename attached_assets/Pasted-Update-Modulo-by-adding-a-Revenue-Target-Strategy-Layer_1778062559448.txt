Update Modulo by adding a Revenue Target Strategy Layer as an add-on to the existing AI Rate calculation.

This should NOT replace the existing Modulo Rate or the existing AI Rate logic.

The goal is to preserve the current pricing algorithm and enhance it with an additional target-aware strategy layer that helps determine whether vacant units should be priced for volume, premium revenue capture, or held neutral based on revenue growth targets, sales velocity, time remaining in the year, and expected revenue impact.

Important architecture:

Street Rate
→ Existing Modulo Rate
→ Existing AI Rate
→ NEW Revenue Target Strategy Layer
→ Final Target-Aware AI Rate
→ Existing Guardrails
→ Stored AI Rate Outcome

Do not delete or overwrite the existing AI Rate logic. Add this as a supplemental module/function that runs after the existing AI Rate is calculated and before the final guardrails are applied.

Name the new layer something like:

calculateRevenueTargetStrategyAdjustment()
calculateTargetAwareAIRate()
or
applyRevenueGrowthStrategyLayer()

The existing AI Rate should remain the starting point. The new layer should calculate a strategic adjustment on top of the existing AI Rate.

Core objective:

The new layer should not blindly increase prices when a campus is behind target. It should determine whether the best strategy is:

1. Volume Driver
   Use lower pricing to accelerate leasing when faster move-ins are likely to create more total revenue.

2. Premium Driver
   Use premium pricing when a unit has strong demand, premium attributes, scarcity, or competitor support, and a higher price does not materially reduce sale probability.

3. Neutral
   Keep the existing AI Rate mostly unchanged when the current rate is already close to optimal.

The new logic should optimize expected revenue, not just price.

A lower price can be correct if it increases the probability and speed of sale enough to improve expected revenue before year-end.

A higher price can be correct if the unit is premium, scarce, or in demand, and the increased rate more than offsets any reduced sale probability.

Step 1: Read revenue target data

Use the existing data structures where available. Read target revenue growth percentage and actual YTD revenue growth by location and service line from:

- revenue_growth_targets
- rent_roll_data
- existing revenue performance services if available

Calculate the following:

Target Growth %
Actual YTD Growth %
Growth Gap % = Target Growth % - Actual YTD Growth %

Projected Year-End Revenue at Current Run Rate
Target Year-End Revenue
Revenue Gap Dollars = Target Year-End Revenue - Projected Year-End Revenue

Months Remaining = number of months remaining in the calendar year
Weeks Remaining = number of weeks remaining in the calendar year

Urgency Score should be a value from 0 to 1.

Suggested formula:

Urgency Score =
clamp(
  (Growth Gap % / max(Months Remaining, 1)) / urgency_divisor,
  0,
  1
)

The urgency divisor should be configurable.

Urgency Score should increase the willingness to take action, but it should not automatically increase prices or automatically discount units.

Step 2: Calculate sales velocity

Calculate sales velocity by:

- location
- service line
- unit type
- room type if available
- care level if available

Use the last 90 days of rent roll history.

Calculate:

- leases per month
- average days to lease
- median days to lease
- average days vacant before lease
- current vacant inventory
- average days vacant for currently vacant units
- estimated base weekly sale probability

Suggested calculation:

Base Weekly Sale Probability =
leases in last 90 days / available unit weeks

If available unit weeks are not reliable, use:

Base Weekly Sale Probability =
1 / average days to lease × 7

Make sure the function handles missing or limited data gracefully.

If there is not enough unit-type history, fall back to service-line history.
If there is not enough service-line history, fall back to campus history.
If there is not enough campus history, use a conservative default.

Step 3: Segment vacant units

For each vacant unit, classify it as:

- Volume Driver
- Premium Driver
- Neutral

Use the following inputs:

- existing AI Rate
- base Modulo Rate
- current street rate
- competitor average rate
- competitor rate gap
- unit days vacant
- average days vacant for similar units
- sales velocity for that unit type
- premium unit type flag
- room attribute score
- occupancy by service line
- revenue gap
- urgency score
- floor and ceiling rate limits
- lead or tour activity if available

Volume Driver criteria:

A vacant unit should be considered a Volume Driver when several of the following are true:

- campus or service line is behind revenue target
- urgency score is moderate or high
- unit has been vacant longer than similar units
- current AI Rate is above competitor average or above market target
- unit has weak or normal attributes
- service line occupancy is below target
- sales velocity is below the pace needed to close the revenue gap
- discounting is likely to improve expected revenue by accelerating lease-up

Premium Driver criteria:

A vacant unit should be considered a Premium Driver when several of the following are true:

- unit has premium attributes
- unit type is scarce
- unit type has strong historical sales velocity
- unit has not been vacant long
- current AI Rate is below competitor average or market-supported pricing
- campus occupancy is healthy
- service line needs rate lift
- higher pricing is likely to improve expected revenue without materially reducing sale probability

Neutral criteria:

A vacant unit should be considered Neutral when:

- current AI Rate is close to optimal
- unit days vacant are normal
- price is close to competitor average
- urgency score is low
- discounting does not improve expected revenue
- premium pricing does not improve expected revenue
- there is not enough confidence to justify a strategic adjustment

Step 5: Estimate sale probability for each candidate rate

For each candidate rate:

Price Change % =
Candidate Rate / Existing AI Rate - 1

Sales Velocity Multiplier =
1 - elasticity_factor × Price Change %

Clamp the Sales Velocity Multiplier to reasonable bounds so it cannot go negative or create extreme results.

Example:

Sales Velocity Multiplier =
clamp(Sales Velocity Multiplier, 0.50, 1.50)

Then calculate:

Adjusted Weekly Sale Probability =
Base Weekly Sale Probability
× Sales Velocity Multiplier
× Days Vacant Factor
× Occupancy Factor
× Competitor Position Factor
× Unit Attribute Factor
× Lead Activity Factor if available

Suggested factors:

Days Vacant Factor:
Increase probability impact for stale units when price is reduced.
Decrease confidence in premium increases for stale units.

Occupancy Factor:
If occupancy is low, favor strategies that improve sale probability.
If occupancy is high, allow more premium pricing.

Competitor Position Factor:
If candidate rate is above competitor average, reduce sale probability.
If candidate rate is below competitor average, increase sale probability.

Unit Attribute Factor:
Premium units can support higher rates better than standard units.

Lead Activity Factor:
If lead or tour volume is strong, premium pricing is more acceptable.
If lead or tour volume is weak, volume pricing may be more appropriate.

Expected Sale Probability by Year-End =
1 - EXP(-Adjusted Weekly Sale Probability × Weeks Remaining)

Expected Weeks to Lease =
1 / Adjusted Weekly Sale Probability

Expected Move-In Date =
Today + Expected Weeks to Lease + average move-in lag

Revenue Months Remaining =
months between Expected Move-In Date and year-end

Expected Revenue =
Expected Sale Probability by Year-End
× Candidate Monthly Rate
× Revenue Months Remaining

Also calculate exit-rate value:

Expected Exit Rate Revenue =
Expected Sale Probability by Year-End
× Candidate Monthly Rate

Step 6: Select the best candidate rate

For each unit, compare the candidate rates against the existing AI Rate.

Existing AI Rate Expected Revenue =
Expected revenue using the existing AI Rate before this new layer

Candidate Incremental Expected Revenue =
Candidate Expected Revenue - Existing AI Rate Expected Revenue

The selected target-aware AI rate should generally be the candidate with the highest total score:

Candidate Score =
Expected Revenue
+ exit_rate_weight × Expected Exit Rate Revenue
+ Target Urgency Bonus
+ Occupancy Health Bonus
- Discount Penalty
- Premium Risk Penalty
- Rate Volatility Penalty
- Low Confidence Penalty

Important rules:

1. A Volume Driver discount should only be recommended if:
   - expected revenue improves versus the existing AI Rate, or
   - occupancy risk is meaningfully reduced and the revenue loss is small and explainable.

2. A Premium Driver increase should only be recommended if:
   - expected revenue improves versus the existing AI Rate, or
   - exit-rate revenue improves without materially reducing sale probability.

3. Neutral units should stay close to the existing AI Rate.

4. Do not allow urgency score by itself to force a discount or price increase.

5. If the best candidate rate is not meaningfully better than the existing AI Rate, keep the existing AI Rate.

Final product logic:

The Revenue Target Strategy Layer should be an enhancement to the existing AI Rate, not a replacement.

The existing AI Rate answers:
"What is the best rate based on the current pricing algorithm?"

The new target-aware strategy layer answers:
"Given our revenue goal, time remaining, sales velocity, and vacant unit mix, should we adjust the AI Rate for volume, premium revenue capture, or leave it neutral?"

The final AI Rate should preserve the current algorithm while adding a smarter revenue-growth strategy on top.
